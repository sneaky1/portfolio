Full-Stack Application Case Study

Chromebook Repair Management System

A web-based application designed to centralize repair tickets, inventory, parts requests, reporting, device images, charges, and service workflows.

View GitHub Repository Back to Projects

Project Overview

The problem

Repair information was spread across disconnected processes, making it difficult to track device status, parts usage, repair history, charges, inventory levels, and reporting.

Goal

Create one reliable system for submitting, assigning, updating, completing, searching, printing, and reporting on Chromebook repairs.

My Role

I designed the workflow, database integration, API, user interface, reporting features, security controls, deployment process, and ongoing enhancements.

Users

The application supports technology staff, repair teams, library staff, and other authorized users involved in device-service operations.

Result

The system replaced fragmented repair tracking with a centralized application that improves visibility, consistency, accountability, and reporting.

Architecture

How the system is built

Front End

Blazor Web Application

Responsive pages and reusable components provide ticket, inventory, reporting, search, print, and administration workflows.

Application Layer

ASP.NET Core REST API

API controllers handle ticket operations, lookup data, image access, inventory, parts requests, reports, suggestions, metrics, and administrative tasks.

Data Layer

SQL Server and Entity Framework Core

Relational data stores repair tickets, images, parts, charges, requests, inventory activity, and reporting data.

Key Features

Core application capabilities

Ticket Management

  • Create and update repair tickets
  • Search open and closed tickets
  • Assign technicians and track status
  • Record repair notes and completed work
  • Print individual tickets and reports

Inventory and Parts

  • Track quantities by location
  • Record parts used during repairs
  • Submit and manage parts requests
  • Identify low-stock inventory
  • Support quantity adjustments and reporting

Reporting

  • Open and closed ticket reporting
  • Damage and charge summaries
  • Repair trends and issue counts
  • Date-based and asset-based reports
  • Dashboard metrics and drill-down views

Files and Images

  • Upload repair images
  • Display ticket image galleries
  • Store image records with ticket relationships
  • Preview and remove selected images
  • Keep production storage paths outside source control

Security

Security and access decisions

  • Role-based access based on authorized user groups
  • API authentication and protected application routes
  • Configuration secrets excluded from Git repositories
  • Sanitized example configuration files for public use
  • Audit logging for important application activity
  • Production server paths and credentials kept outside source code
  • Public repository stripped of organization-specific data and branding

Technology Stack
